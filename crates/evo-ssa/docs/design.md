# evo-ssa Design

## Scope

`evo-ssa` is an experimental exact SSA engine for evolutionary systems. The intended model shape is a small fixed set of reaction families, such as birth, death, and mutation, with a dynamic and potentially large set of concrete channels inside those families. A mutation can create a new species at runtime; the scheduler should update only the affected concrete channels rather than rebuild or resample the whole system.

## Public Layers

The crate exposes two model APIs.

`EvolvingModel` is the dynamic API. It supports arbitrary channel keys and payloads, and is scheduled by `DirectScheduler` or `DenseDirectScheduler`. This is the easiest API to use while prototyping dynamic channel creation.

`StaticDirectModel` plus `StaticReactionFamily` is the static-family API. It is for models whose reaction families are known at compile time and whose concrete channels can be addressed by dense local indices. This API backs `StaticDirect`, `FamilyDirect`, `FamilyNrm`, and `StaticFamilyNrm`.

## Update Contract

After a reaction fires, the model reports exactly which concrete channels need their cached propensity or clock updated. Static reaction families do this through `StaticUpdateSink`:

- `recompute(local_channel)` updates a channel in the current family.
- `recompute_family(family, local_channel)` updates a channel in an explicit family.
- `recompute_global(channel)` supports flattened static direct schedulers.

Hot family-separated schedulers use `(family, local_channel)` updates internally. This avoids converting through a flattened global channel id after every event. `StaticFamilyNrm<M, Families, UPDATE_CAP>` writes updates into a fixed-capacity stack buffer whose capacity is selected by the caller.

## Schedulers

`DirectScheduler` is a cached Direct Method scheduler for arbitrary channel keys. It stores one cached propensity per active channel and samples from a segment tree. Channel insert, remove, and recompute are `O(log M)`.

`DenseDirectScheduler` is the same method for dense `usize` channel ids. It avoids the hash map used by `DirectScheduler`.

`StaticDirect` flattens all compile-time reaction families into one segment tree. It is mostly useful as a static-dispatch baseline.

`FamilyDirect` keeps one segment tree per reaction family. It samples the family first, then the local channel. This is the preferred Direct Method baseline for fixed-family evolutionary models.

`FamilyNrm` is a generic family-separated Next Reaction Method scheduler. It stores one indexed min-heap per family and one clock per concrete channel. It supports the same macro-generated tuple arities as `FamilyDirect`.

`StaticFamilyNrm` is the optimized static-family hot path. It is generic over a tuple of reaction families and a const update-buffer capacity. Tuple implementations are generated by macro for small arities, so the scheduler is not tied to birth/death/mutation or to exactly three families.

## NRM Clock Semantics

The NRM implementation assumes piecewise-constant propensities between events. Every channel stores its current propensity and scheduled firing time. When a channel fires, it draws a fresh exponential clock. When a non-fired channel's propensity changes, the scheduler applies the Gibson-Bruck residual update:

```text
new_time = now + old_propensity / new_propensity * (old_time - now)
```

Channels with zero propensity are removed from the heap and keep an inactive clock. If a later update gives them positive propensity, they receive a fresh scheduled time.

## Benchmark Workloads

The benchmark suite keeps a small set of comparisons:

- one-species birth/death
- many-species birth/death
- mutation creating new species
- trait mutation birth/death with random child rates

The trait mutation benchmark includes a hand-written family-separated NRM as an upper-bound reference. The reviewable framework path is `StaticFamilyNrm`; prior experiments with a flat heap and a SoA state layout were removed because they either measured an algorithm shape we do not intend to use or were slower for sparse NRM updates.

## Current Design Tradeoffs

`StaticFamilyNrm` is intentionally static rather than fully dynamic. It keeps the general reaction-family API but lets the compiler specialize the scheduler for a supported tuple arity. If production models need more families than the current macro list supports, extend the macro invocation rather than adding a workload-specific scheduler type.

The static family API still requires reaction families to name their affected channels. This is deliberate: the framework should not infer dependencies by scanning species or channels. If a model has broad all-to-all dependencies, that cost should be explicit in the update emission logic.
